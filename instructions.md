## PART 11 — WEBSITE STRUCTURE

### Design Philosophy
**Terminal × Mission Control × Engineering Log**

The aesthetic is a dark engineering OS dashboard — not a lifestyle planner. It feels like software running a real mission. Key design decisions:
- Phosphor green (#00e5a0) as the primary accent — "system active" signal
- Amber (#ffb020) for warnings and time pressure
- Red (#ff4444) for hard deadlines and urgency
- All typography: `Geist Mono` (monospace body) + `Bebas Neue` (display headers)
- Scanlines and grid overlays for terminal atmosphere
- No rounded corners except 2px radius
- No drop shadows except glow effects on active/current states

### File Structure
```
sprint-os/
├── index.html        ← Dashboard
├── daily.html        ← Daily schedule
├── courses.html      ← Course checklist
├── milestones.html   ← Milestone tracker
├── savings.html      ← Savings + ETA
├── roadmap.html      ← 18-month roadmap
├── style.css         ← Shared design system
├── data.js           ← All data + shared utilities
├── update.js         ← Real-time engine + Google sync
├── README.md         ← GitHub Pages deployment guide
└── instructions.md   ← This file
```

### Page Descriptions

#### `index.html` — Dashboard
The mission control overview. Shows:
- Live greeting (device-time-aware: good morning/afternoon/evening)
- Daily priority message (auto-updated from current phase + countdown)
- `▶ NOW` and `⏭ NEXT` schedule blocks (live, updates every second)
- Three countdown chips: edX expiry / IELTS exam / OU application
- Five stat cards: savings %, courses done, plan month, milestones, pace tier
- 18-month timeline bar with live current-month indicator
- Google sync bar (sign in, schedule today, add milestone events)
- IELTS status panel + score logger modal
- Quick action links to all other pages

#### `daily.html` — Daily Schedule
The operational daily view. Shows:
- **Order/Free toggle**: persisted per calendar date, resets at midnight
- **Pre/Post-IELTS toggle**: persisted in localStorage, manually switched after exam
- Live tier display (from savings state)
- Intensity meter (animated bar)
- Full day schedule with `▶ NOW` indicator (updates every second via `U.Ticker`)
- Weekly grid summary
- "Add to Calendar" button → calls `U.Cal.addTodaySchedule()`

#### `courses.html` — Course Queue
The full course checklist. Shows:
- Overview stat cards (done / pending / drop / progress %)
- 4 collapsible phase accordions (edX → LinkedIn → Coursera → Udemy)
- Each accordion has: platform status badge, progress bar, grouped course list with month tags
- Each course row: checkbox, priority star (★) or drop marker (✗), name, progress %, platform tag
- State persisted via `getCDone()` / `setCDone()` and synced to Drive

#### `milestones.html` — Milestone Tracker
The accountability checklist. Shows:
- SVG progress rings (overall, urgent, priority)
- Category progress bars (critical, priority, normal, savings)
- IELTS score logger form with individual band inputs (auto-calculates overall)
- Full milestone list grouped by category (Critical → Priority → Normal → Done)
- Each milestone: animated checkbox, type badges, horizon tag
- `U.toggleMilestone()` handles all state + Drive sync + Calendar event creation

#### `savings.html` — Savings Tracker

#### `roadmap.html` — 18-Month Roadmap

### JavaScript Architecture

#### `data.js` — Shared Data Layer
Everything that is static data or a pure utility function:
- `D` — date constants (start, edx, ielts, ouApp, today)
- `S` — savings state object with localStorage persistence
- `TIERS` — 4 pace tier objects
- `TL_EVENTS` — timeline event labels by month number
- `MILESTONES` — array of 17 milestone objects
- `PHASES` — 4 platform phases, each with groups and course arrays
- `getMDone()` / `setMDone()` — milestone state persistence
- `getCDone()` / `setCDone()` — course state persistence
- `flatCourses()` — flatten all phases/groups into a single indexed array
- `getTier()` — returns current tier from `S.monthly`
- `buildSchedule(order, preIelts, tier)` — schedule builder, returns array of blocks
- `buildNavHTML(activePage)` — generates full nav HTML with live data
- `buildFooterHTML()` — generates footer
- `buildTimelineHTML()` — generates 18-month timeline bar HTML
- `daysUntil(d)` — days until a date
- `planMonth()` — current plan month (1–18)

#### `update.js` — Real-Time Engine
Everything that involves time, sync, or mutation:

**`U.Clock`** — device clock wrapper
- `now()`, `hour()`, `minute()`, `dow()`, `hhMM()`, `isWeekend()`, `isSaturday()`
- `currentBlock(schedule)` — returns the schedule block active right now

**`U.Phase`** — IELTS phase detection
- `isPreIelts()` — reads from localStorage
- `setIeltsTaken(score)` — persists exam result, switches mode
- `ieltsTaken()`, `ieltsScore()`, `ieltsTakenDate()`

**`U.Routine`** — daily routine resolver
- `hasOrder()` / `setOrder(v)` — persisted per calendar date key
- `todaySchedule()` — builds the correct schedule for today
- `currentBlock()` / `nextBlock()` — current and upcoming schedule items
- `greeting()`, `dailyPriority()` — context-aware text

**`U.Drive`** — Google Drive sync
- `initGapi()` — initialises the Google API client
- `save()` — saves snapshot to Drive appdata (with localStorage fallback)
- `load()` — loads snapshot from Drive and applies to localStorage
- `buildSnapshot()` — serialises all state to a JSON object
- `applySnapshot(snap)` — restores state from JSON
- `signIn()`, `signOut()`, `isSignedIn()`

**`U.Cal`** — Google Calendar integration
- `addTodaySchedule()` — creates calendar events for today's non-rest blocks
- `addMilestoneEvent(milestone)` — adds a milestone deadline to calendar
- `addIeltsExam()` — adds IELTS exam date with reminders
- `addEdxExpiry()` — adds edX expiry deadline

**`U.Ticker`** — live clock (1-second interval)
- Updates `#live-clock`, `#cnt-edx`, `#cnt-ielts`, `#cnt-ou`
- Calls `_highlightCurrentBlock()` — adds `sched-now` class + `▶ NOW` badge
- Fires `routine:refresh` event at midnight

**`U.AutoSave`** — debounced save manager
- `touch()` — queues a save (2s debounce)
- `flush()` / `now()` — immediate save

**`U.MilestoneWatcher`** — auto-completion logic
- Checks if IELTS score ≥ 7.0 → auto-completes IELTS milestone
- Checks if savings ≥ 5,000 / 10,000 EUR → auto-completes savings milestones

**`U.init()`** — bootstrap (called on every page)
- Loads local state, starts ticker, runs routine logic, inits Google API

### State Architecture

All state is stored in `localStorage` and optionally mirrored to Google Drive.

| Key | Type | Description |
|-----|------|-------------|
| `s_bdt` | number | Current savings in BDT |
| `s_monthly` | number | Monthly saving amount in BDT |
| `m_done` | JSON array | boolean[] parallel to MILESTONES |
| `c_done` | JSON array | boolean[] parallel to flatCourses() |
| `pre_ielts` | 'true'/'false' | Current phase |
| `ielts_taken` | 'true'/'false' | Whether exam has been sat |
| `ielts_taken_date` | ISO string | Date exam was taken |
| `ielts_score` | number string | Overall band score |
| `cal_events` | 'true'/'false' | Whether to create Calendar events |
| `cal_reminders` | 'true'/'false' | Whether Calendar events have reminders |
| `order_day_<date>` | 'true'/'false' | Whether today has order work (resets daily) |
| `savings_log` | JSON array | Last 50 savings log entries |
| `sprint_snapshot` | JSON | Full state backup for Drive fallback |
| `goal_done` | JSON object | Roadmap goal completion by key `period_goal` |
| `rp_open` | JSON object | Which roadmap period accordions are open |

**Drive state file:** `sprint-os-state.json` in Google Drive appdata folder (private to this app only).

## PART 12 — EVENTS SYSTEM

Pages communicate via `document.dispatchEvent`:

| Event | Fired when | Listened by |
|-------|-----------|-------------|
| `ui:refresh` | General update needed | All pages |
| `savings:updated` | Savings BDT or monthly changed | index, savings |
| `milestones:updated` | Any milestone toggled | index, milestones |
| `courses:updated` | Any course toggled | courses |
| `routine:refresh` | Midnight rollover | daily, index |
| `state:loaded` | Drive load complete | All pages |
| `gapi:auth` | Sign-in state changed | index |
| `gapi:ready` | Google API initialised | Any page |
| `sync:status` | Drive save/load status | All pages (sync-status element) |
| `phase:suggest-switch` | Device date past IELTS target | index (shows toast) |
| `routine:saturday-mock` | It's Saturday + pre-IELTS | index (shows toast) |
| `routine:edx-critical` | edX ≤ 3 days away | index (shows toast) |
| `routine:tier` | Tier computed for this session | Any page |
| `routine:income-gap` | 1st of month + savings below target | Any page |

## PART 14 — HOW TO USE THIS FILE WITH AN AI ASSISTANT

If you are an AI assistant reading this to continue improving Sprint OS, here is the protocol:

1. **Read this file first.** It contains everything you need to know about Anway, the mission, the data, and the code structure.
2. **Read `data.js` next.** All data structures and the schedule builder are there.
3. **Read `update.js` next.** The real-time engine and sync architecture are there.
4. **Ask which page or feature to work on.** Do not assume.
5. **Preserve the design system.** Every colour, font, and spacing token is in `style.css` as CSS variables. Never hardcode colours in new components.
6. **Preserve state key names.** The `localStorage` key names in Part 12 are canonical. Changing them breaks existing saved data.
7. **Never remove the `buildNavHTML()` call from any page.** It generates the navigation which must remain consistent.
8. **Any new page must:**
   - Include `<script src="data.js"></script>` and `<script src="update.js"></script>`
   - Call `U.init()` after the gapi script loads
   - Call `buildNavHTML('newpage.html')` for the nav
   - Call `buildFooterHTML()` for the footer
   - Listen to `document.addEventListener('ui:refresh', ...)` and `document.addEventListener('state:loaded', ...)`
9. **The tone is engineering-grade.** No motivational language, no emojis in UI text, no padding. Direct, structured, technical.
10. **Mobile is important.** Anway uses his phone a lot. Any new component must have responsive CSS.
